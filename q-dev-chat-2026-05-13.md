## Step 1 — Complete JSON Schema Design

---

### Core Principles from your answers:

- Every question is **free-form text** but has a **question number**
- Each question is **pre-split into sub-checks** — each sub-check is its own entry with its own mark
- Sub-checks run independently, combined total reported at the end
- Table checks allow **tolerance mode** — text/formatting can pass even if merge failed
- Partial marks are natural — each sub-check is 1 unit, they add up

### Tolerance Mode for Tables

When `"tolerance": true` on a `cell_text` check, the engine will:
1. First try the exact cell `(row, col)`
2. If empty, scan the same row for the text in any cell
3. If found elsewhere, mark text as **passed** but log a tolerance warning
4. Merge check remains a **separate sub-check** and fails independently

### Summary of Design Decisions

| Decision | Reason |
|---|---|
| Sub-checks are flat, not nested | Simpler engine dispatch, easier to extend |
| `tolerance: true` on table checks | Handles "visually merged" edge case fairly |
| `unit` always explicit in expected | Avoids cm/pt/twip conversion bugs |
| Locators are separate from domain | Same locator system works across all domains |
| `feedback_pass` / `feedback_fail` optional | LLM can generate these, or hardcode them |
| question_number is free-form string | Matches real marksheets like "1.1.1", "Q2a" |

**Office Theme Color Names — Yes, fully possible**

Word stores theme colors in two places inside the `.docx` zip:

1. `word/theme/theme1.xml` — defines the theme palette (accent1 through accent6, dk1, lt1 etc.)
2. The run XML uses `w:color` with either:
   - `w:val="4472C4"` (resolved RGB) — direct color
   - `w:themeColor="accent1"` + `w:themeTint="99"` or `w:themeShade="BF"` — theme reference

So "Blue, Accent 2, 25% Lighter" breaks down as:

```
themeColor = accent2
themeTint  = BF   (hex for ~75% = 25% lighter means 75% tint)
```

The mapping table looks like this:

```python
THEME_COLOR_NAMES = {
    "dk1":     "Dark 1",
    "lt1":     "Light 1", 
    "dk2":     "Dark 2",
    "lt2":     "Light 2",
    "accent1": "Accent 1",
    "accent2": "Accent 2",
    "accent3": "Accent 3",
    "accent4": "Accent 4",
    "accent5": "Accent 5",
    "accent6": "Accent 6",
}

TINT_SHADE_NAMES = {
    # Tints (lighter) — themeTint values
    "FF": "",            # 0% = base color
    "E6": "10% Lighter",
    "CC": "25% Lighter", # ← "Blue, Accent 2, 25% Lighter"  
    "99": "50% Lighter",
    "66": "60% Lighter",  
    "33": "80% Lighter",
    # Shades (darker) — themeShade values
    "BF": "25% Darker",
    "7F": "50% Darker",
    "40": "75% Darker",
}
```

So the resolver function would work like this:

```python
def resolve_color_name(w_color_element, theme_map):
    theme  = w_color_element.get(WN + "themeColor")  # e.g. "accent2"
    tint   = w_color_element.get(WN + "themeTint")   # e.g. "CC"
    shade  = w_color_element.get(WN + "themeShade")  # e.g. "BF"
    hex_val = w_color_element.get(WN + "val")        # e.g. "4472C4"

    if theme:
        base_name = THEME_COLOR_NAMES.get(theme, theme)
        modifier  = TINT_SHADE_NAMES.get((tint or shade or "FF").upper(), "")
        full_name = f"{base_name}, {modifier}".strip(", ")
        # e.g. "Accent 2, 25% Lighter"
        return full_name, hex_val
    
    # No theme — just return hex and try to match known colors
    return resolve_hex_name(hex_val), hex_val
```

And for the feedback it would produce:

```
Font color was 'Accent 2, 25% Lighter (#9DC3E6)' — expected 'Accent 2, 25% Lighter (#9DC3E6)'  ✅
Font color was 'Dark Red (#C00000)' — expected 'Accent 2, 25% Lighter (#9DC3E6)'  ❌
```

---

**One important note on theme colors:**

When a student applies "Blue Accent 2, 25% Lighter" Word sometimes stores the **resolved hex** (`9DC3E6`) instead of the theme reference, depending on the Word version. So the checker needs to handle **both**:

1. Match by `themeColor` + `themeTint/themeShade` — exact theme match
2. Match by resolved hex value as fallback — covers older Word versions
